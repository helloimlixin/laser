## Failed 256 Sweep

### `/scratch/xl598/runs/celebahq256_patch_sweep`

These runs did not complete. The matching SLURM `.err` logs in the workspace show the same failure across all four jobs:

- `celebahq256_patch_d4_a1536_k12_50433254.err`
- `celebahq256_patch_d4_a1536_k16_50433255.err`
- `celebahq256_patch_d4_a3072_k12_50433256.err`
- `celebahq256_patch_d4_a3072_k16_50433257.err`

Failure mode:

```text
RuntimeError: expected scalar type Float but found BFloat16
```

The stack points at stage-1 validation SSIM inside `proto.py`:

- `_batch_ssim(...)`
- `F.conv2d(y, kernel, padding=radius, groups=channels)`

So this family is not “bad performance”; it is a concrete BF16 validation bug in the raw 256 sweep path.

## Practical Read

If you need a stable reference from saved artifacts today:

1. Use `laser_fast100` as the strongest completed `128 x 128` baseline.
2. Use `laser_patch100/20260318_083442` as the completed overlapping-patch `128 x 128` reference.
3. Use `celebahq256_patch_coeff_compare` as the only completed and decision-grade `256 x 256` patch family.
4. Do not interpret `celebahq256_patch_sweep` as a model comparison; it failed on the BF16 SSIM validation path.
5. Treat `celebahq256_patch_sweep_packed` and `celebahq256_patch_tuned_sweep` as partial historical runs, not completed baselines.
